# FoundIR 项目结构问题总结（2026-04-20）

## 目的
- 记录当前 `algorithm/FoundIR` 项目在工程结构、训练入口、测试入口、数据组织和可维护性上的主要问题。
- 为后续“整理 FoundIR 代码、重建可控实验底座”提供依据。

## 总体判断
- `FoundIR` 的核心建模思路可以学习，尤其适合作为“理解条件扩散恢复”的参考项目。
- 但它当前仓库状态并不适合直接作为本科毕设主线工程使用。
- 主要问题不是单一 bug，而是：
  - 入口脚本实验化严重
  - 参数管理混乱
  - 训练/测试逻辑硬编码过多
  - 数据流与目录约定强耦合
  - 条件分支代码存在明显错误
  - 工程边界不清晰

一句话概括：

`FoundIR` 更像“作者自己实验时能跑的研究代码”，而不是“适合长期维护和二次开发的项目骨架”。

---

## 一、项目结构层面的问题

### 1. 仓库边界不清晰
- 主体代码在：
  - `train.py`
  - `test.py`
  - `src/model.py`
  - `data/combined_dataset.py`
- 但仓库中还混入了：
  - `specialist_model/`
  - `dataset/`
  - `premodel/`
  - `assets/`
  - `eval/`
- 这些目录既包含参考资源，也包含实际代码和权重，角色边界不明确。

### 2. 单文件承担过多职责
- `src/model.py` 既包含：
  - 网络结构定义
  - 扩散过程定义
  - 采样逻辑
  - 训练器 `Trainer`
  - patch 切分与合并逻辑
  - 图像保存逻辑
- 这导致单文件过于臃肿，理解和修改成本高。

### 3. 训练、测试、评估未模块化
- 训练入口、测试入口、指标评估彼此分散。
- 没有统一配置层，也没有统一的实验管理层。
- 很多逻辑靠“改脚本内容”而不是“传参数”切换。

---

## 二、训练入口层面的问题

### 1. `train.py` 参数设计混乱
文件：
- `algorithm/FoundIR/train.py`

问题表现：
- 同时使用 `argparse` 和 `sys.argv`。
- `sampling_timesteps` 不是通过正式参数定义，而是直接读 `sys.argv[1]`。

对应代码：
- `train.py:29-32`

问题本质：
- 参数来源不统一。
- 后续扩展训练配置时容易出现行为不透明、调用不一致的问题。

### 2. GPU 环境硬编码
对应代码：
- `train.py:24`

问题表现：
- 直接写死：
```python
os.environ['CUDA_VISIBLE_DEVICES'] = '0,1,2,3,4,5,6,7'
```

风险：
- 在单卡、双卡、不同机器、容器环境下都不稳。
- 训练脚本不具备可移植性。

### 3. 训练关键超参数写死在脚本正文
例如：
- `train_num_steps = 500000`
- `image_size = 512`
- `results_folder = "./ckpt_single_multi"`
- `objective = 'pred_res'`
- `delta_end = 1.4e-3`

问题本质：
- 训练配置没有抽象成独立配置文件或完整 CLI 参数。
- 实验可复现性差。
- 多组实验对比时需要频繁手改源码。

### 4. 训练脚本不像“通用入口”，更像“当前实验快照”
- `train.py` 更像作者当时跑某个实验时留下的脚本。
- 缺少：
  - 正式实验命名机制
  - 输出目录规范
  - 统一 resume 参数
  - 可追踪的配置落盘

---

## 三、测试入口层面的问题

### 1. `test.py` 也是实验脚本，不是通用推理入口
文件：
- `algorithm/FoundIR/test.py`

问题表现：
- 代码里直接区分：
  - “For our testset”
  - “For your own data”
- 通过手动注释切换，而不是通过参数切换。

对应代码：
- `test.py:40-44`

这说明：
- 测试脚本依赖人工改代码。
- 不适合作为稳定系统中的推理入口。

### 2. checkpoint 加载写死
对应代码：
- `test.py:99`

问题表现：
```python
trainer.load(2000)
```

影响：
- 测试流程绑定某个固定 milestone。
- 不支持灵活指定权重。
- 对后续实验管理极不友好。

### 3. 测试参数同样大量写死
例如：
- `sampling_timesteps = 4`
- `image_size = 1024`
- `results_folder = 'premodel'`
- `train_batch_size = 1`

问题本质：
- 推理行为难以复用。
- 切换数据集、切换采样步数、切换输出目录都要改代码。

### 4. 测试脚本布尔参数设计不规范
对应代码：
- `test.py:16`

问题表现：
```python
parser.add_argument('--no_flip', type=bool, default=True, ...)
```

风险：
- CLI 中 `type=bool` 的行为很不稳，容易产生“传了但不符合预期”的问题。

## 四、数据层面的问题

### 1. `CombinedDataset` 逻辑耦合重、分支混乱
文件：
- `algorithm/FoundIR/data/combined_dataset.py`

问题表现：
- `task == None` 和 `task == 'meta_info'` 两种路径混在同一个类中。
- 路径加载、配对逻辑、增强、resize、部分工具函数都混在一起。

### 2. 条件分支结构不清晰
对应代码：
- `combined_dataset.py:59-84`

问题描述：
- `if task == None`
- `if task == 'meta_info' and opt.meta is not None`
- `else`

这里的 `else` 是挂在第二个 `if` 上的，不是和第一个 `if` 配对，结构非常绕。

风险：
- 代码虽然在某些路径下能跑，但后续扩展模式时容易踩未初始化变量和逻辑歧义。

### 3. 数据类中残留大量与当前主流程无关的工具函数
例如：
- `load_flist`
- `cv2equalizeHist`
- `get_patch`
- `pad_img`
- `split_image`
- `merge_patches`

这些函数有些并未真正服务当前主流程，有些甚至依赖未定义成员。

这说明：
- 数据类承担了过多责任。
- 存在历史遗留代码堆积。

### 4. 数据集约定强耦合
- 默认目录约定依赖：
  - `LQ`
  - `GT`
  - meta 文件内容格式
- 缺少更清晰的数据接口抽象。

结果是：
- 一旦换自己的数据组织方式，就要频繁改源码。

---

## 五、Trainer 层面的问题

### 1. `Trainer` 中有明显逻辑错误
文件：
- `algorithm/FoundIR/src/model.py`

对应代码：
- `model.py:1214`
- `model.py:1286`

问题代码类似：
```python
if 'combined' or 'all' in results_folder:
```

这在 Python 中恒为真。

实际后果：
- 该条件分支并没有按作者意图工作。
- 后面的 `elif 'paired' ...` 分支事实上不会被正常走到。

这是“代码风格差”之外的真实行为错误。

### 2. `Trainer` 依赖 `results_folder` 字符串决定数据流
例如：
- 根据 `results_folder` 是否包含 `'combined'`、`'all'`、`'paired'` 来决定 dataloader 逻辑。

这本质上是：
- 用输出目录名驱动训练行为。

问题非常严重：
- 配置语义和路径命名耦合。
- 改个文件夹名字都可能改训练行为。

### 3. 训练器职责过重
`Trainer` 不只负责训练，还承担了：
- 保存与加载 checkpoint
- EMA 管理
- 测试
- patch 切图与拼图
- 图像保存

这导致：
- 代码边界混乱
- 很难单独测试某个环节
- 修改一个功能容易影响别的功能

### 4. 测试逻辑也写在 `Trainer` 内部
- `test()` 中包含大图切块推理、保存路径构造、输出写盘等逻辑。
- 这使得推理流程很难被单独复用成简洁的 inference API。

---

## 六、评估与实验管理层面的问题

### 1. 评估逻辑残缺
- `test()` 中本来存在 PSNR/SSIM 统计逻辑，但大段被注释掉。

说明：
- 项目里存在“曾经写过，但目前不是稳定工作状态”的评估路径。

### 2. 缺少统一实验配置记录
- 训练后没有看到标准化保存：
  - 完整参数
  - git 版本
  - 数据集信息
  - 采样配置

这会导致：
- 之后难以严格复现实验。

### 3. 缺少清晰的输出目录规范
- `ckpt_single_multi`
- `premodel`
- `results`

这些目录名更像临时命名，而不是长期维护项目中的规范命名。

---

## 七、代码质量层面的典型症状

### 1. 命名不规范
- `parsr_args()` 拼写错误。
- `adap`、`gt`、`task`、`condition` 等命名语义不统一。

### 2. 注释风格偏“作者自用”
- 有些注释是说明实验场景。
- 有些是直接留的切换提示。
- 缺少面对维护者的清晰文档。

### 3. 大量硬编码
- device
- 路径
- 权重步数
- 采样步数
- 图像大小
- 训练总步数

### 4. 死代码和遗留代码偏多
- 注释掉的逻辑很多。
- 类中保留了不一定被当前主流程使用的方法。

---

## 八、结论

### 1. FoundIR 当前“不适合作为未经整理的主线工程”
原因不是它的扩散思路不行，而是：
- 当前代码组织不适合直接长期维护
- 也不适合直接作为系统毕设的稳定底座

### 2. FoundIR 仍然有保留价值
它的价值主要在：
- 帮助理解条件扩散恢复任务的实现思路
- 作为你后续“整理后自用版本”的起点
- 作为对照路线或学习路线

### 3. 如果继续用 FoundIR，第一步不是继续堆实验，而是先整理工程
优先级建议：
1. 重做训练入口
2. 重做测试入口
3. 抽离配置
4. 重构数据集类
5. 修正 Trainer 中的条件分支错误
6. 拆分 `src/model.py` 中过重职责

---

## 九、建议的整理方向

### 第一阶段：最小可控化
- 修复明显逻辑错误。
- 把训练和测试入口参数化。
- 去掉依赖目录名驱动行为的写法。
- 支持显式指定：
  - 数据路径
  - meta 文件
  - checkpoint
  - 输出目录
  - 采样步数

### 第二阶段：工程解耦
- 将 `src/model.py` 拆分为：
  - 网络结构
  - 扩散过程
  - 训练器
  - 推理工具
- 简化 `CombinedDataset`。

### 第三阶段：系统化整合
- 接入你主仓库统一的数据格式。
- 建立统一日志、评估、结果输出体系。
- 使 FoundIR 成为“系统中的一个可切换模型”，而不是一个孤立仓库。

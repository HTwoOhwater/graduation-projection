# JiT 阶段A训练测试归档（截至 2026-04-19）

## 归档目的
- 汇总 JiT 阶段A“最小可跑条件化”从方案确定、代码改造、冒烟验证、四卡训练到推理测试的完整过程。
- 为后续阶段B/阶段C推进提供统一上下文，避免信息散落在多份记录中。

## 阶段定位
- 当前阶段为 `阶段A：最小可跑条件化`。
- 目标是将 JiT 从类别条件生成改造成图像条件重建模型的最小闭环版本。
- 本阶段不追求最优恢复效果，优先验证：
  1. 配对数据可读取。
  2. 模型可接收条件图 `y_cond`。
  3. 训练可稳定跑通。
  4. 推理路径可生成与条件图相关的重建结果。

## 设计边界
- 保留 JiT 原生核心机制不变：
  - 纯 ViT 像素空间架构。
  - 监督目标仍为速度场 `v`。
  - 不改为 `pred_res` 或 `pred_x0`。
  - 采样器仍使用 JiT 原生 ODE 路径（Euler/Heun）。
- 条件注入采用阶段A最小方案：
  - 输入级通道拼接 `cat(z_t, y_cond)`。
- 当前未实现：
  - 条件邻域初始化。
  - token 级 In-Context 条件融合。
  - 面向重建任务的采样路径重构。

## 代码改造归档

### 1. 数据层
- 新增 `algorithm/JiT/util/paired_dataset.py`
- 支持两种配对模式：
  - `stem`
  - `meta`
- 输出格式：
  - `(y_cond, x_gt, label)`

### 2. 模型接口层
- 修改 `algorithm/JiT/denoiser.py`
  - `forward(x, labels=None, y_cond=None)`
- 修改 `algorithm/JiT/model_jit.py`
  - `forward(x, t, y=None, y_cond=None)`
- paired 模式下默认关闭类别条件，仅在 `--use_paired_label_condition` 开启时才使用 label。

### 3. 条件注入层
- 将 `z_t` 与 `y_cond` 在通道维拼接：
  - `x_in = cat(z_t, y_cond)`
- paired 默认输入通道数：
  - `6`
- 输出通道数：
  - `3`

### 4. 训练层
- 修改 `algorithm/JiT/main_jit.py`
  - 新增参数：
    - `--dataset_mode`
    - `--train_split`
    - `--lq_dirname`
    - `--gt_dirname`
    - `--pairing_mode`
    - `--pair_meta`
    - `--image_channels`
    - `--use_paired_label_condition`
    - `--max_train_steps`
- 修改 `algorithm/JiT/engine_jit.py`
  - 训练循环适配 paired batch。
  - 加入 `tqdm` 进度条。

### 5. 损失函数可配化
- 修改 `algorithm/JiT/main_jit.py`
  - 新增：
    - `--loss_type {l2,l1,l1_l2}`
    - `--loss_l1_weight`
    - `--loss_l2_weight`
- 修改 `algorithm/JiT/denoiser.py`
  - 保持监督目标为速度场 `v`。
  - 支持：
    - `L2`
    - `L1`
    - `加权 L1 + L2`

### 6. 推理补线
- 修改 `algorithm/JiT/denoiser.py`
  - 新增 `generate_paired(y_cond, labels=None)`。
  - 新增 paired 版 Euler/Heun 采样步骤。
- 新增 `algorithm/JiT/infer_paired.py`
  - 支持载入 EMA checkpoint。
  - 支持对 `datasets/{split}` 中的 paired 样本做恢复。
  - 支持保存：
    - `input`
    - `pred`
    - `gt`
    - `triptych`
- 兼容 PyTorch 2.6：
  - `torch.load(..., weights_only=False)`

### 初步结论
- 当前阶段A结果可以认定为：
  - 训练闭环已打通
  - paired 推理闭环已打通
  - 模型对条件图存在一定响应
- 但恢复效果整体不强，现阶段只能算：
  - `可运行`
  - `可比较`
  - `不够好`

### 效果不强的主要原因判断
1. 推理仍从纯噪声初始化，缺少“条件邻域初始化”。
2. 条件注入仍是最弱的通道拼接方案。
3. 阶段A目标是验证可跑，不是追求恢复性能上限。
4. 当前只做了少量样本测试，尚未形成整套 test 集统计结果。

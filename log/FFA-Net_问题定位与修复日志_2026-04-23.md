# FFA-Net 问题定位与修复日志

日期：2026-04-23

## 1. 背景

在本地数据集上对 `FFA-Net` 进行迁移训练后，推理结果出现明显异常：

- 色彩表现怪异
- 部分 checkpoint 的结果比预训练权重更差
- 训练到 `5000` 步时看起来像“卡死”

因此对 `FFA-Net` 的训练链路、验证逻辑、checkpoint 行为进行了专项排查。

## 2. 已确认的问题

### 2.1 `5000` 步“卡死”并不是真卡死

定位结果：

- `main.py` 中默认 `eval_step=5000`
- 每到 `5000` 步就会进入整轮验证
- 本地 `valid` 集规模较大
- 原始验证逻辑没有进度输出，因此表现为长时间无日志，容易误判为卡死

处理结果：

- 为验证过程增加了显式进度打印
- 新增 `--eval_print_freq`
- 在根目录 `README.MD` 中补充了 `--eval_limit` 和 `--eval_print_freq` 的推荐用法

## 3. checkpoint 对比结果

对以下权重分别进行了同批样本推理对比：

- `model-5000.pt`
- `model-15000.pt`
- `model-20000.pt`
- `model-30000.pt`
- `model-last.pt`

结论：

- `model-last.pt` 在当前实验中反而是相对更好的一个
- 但整体仍存在训练后性能退化现象
- 说明问题不在单纯“最后一个 checkpoint 存错了”，而在训练过程本身

## 4. 非数据读取问题

已经排除“图像加载/保存逻辑错误导致颜色异常”的可能性，依据如下：

- 同一套 `test.py` 推理逻辑下，原始 `ITS/OTS` 预训练权重输出正常
- 仅本地微调后的 checkpoint 出现明显异常

因此可以判断：

- 推理输入输出链路基本正常
- 问题主要来自训练阶段

## 5. 核心训练问题

排查 `main.py` 训练循环后，发现一个关键错误：

原始代码使用：

```python
x, y = next(iter(loader_train))
```

这意味着：

- 每一步都会重新创建一个新的 dataloader iterator
- 每一步都只取一个“新 iterator 的第一批”
- 训练并没有正常顺序遍历整个 dataloader

这会直接导致：

- 数据遍历方式异常
- 训练统计不稳定
- checkpoint 表现不可预测
- 容易出现迁移训练把颜色和整体表现训坏的情况

## 6. 修复内容

已经修复训练循环，改为正常迭代方式：

- 先创建 `train_iter = iter(loader_train)`
- 每步使用 `next(train_iter)`
- 遇到 `StopIteration` 后再重建 iterator

这样现在的 `FFA-Net` 训练：

- 能够正常遍历 dataloader
- 不会每一步重复取“第一批”
- 训练行为和 loss 统计更可信

## 7. 学习率结论

经过重新训练与对比，当前已确认：

- 之前 `FFA-Net` 效果异常的主要原因之一是学习率过大
- 在修正训练循环之后，使用更小学习率重新微调，结果明显更好

当前结论为：

- 不是数据读取错误
- 不是推理脚本本身有根本问题
- 主要是“训练循环错误 + 学习率过大”共同导致模型训练失稳

## 8. 当前建议

后续继续跑 `FFA-Net` 时，建议采用以下策略：

- 优先从 `ITS/OTS` 预训练权重迁移
- 学习率优先使用 `1e-5` 或 `2e-5`
- `num_workers` 先保持 `0`
- 验证阶段使用 `--eval_limit 100`
- 不只看 `model-last.pt`，同时观察中间 checkpoint
